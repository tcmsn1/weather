# 物联网天气平台：系统是怎么实现的？（小白友好版）

本文用**通俗比喻**配合**技术说明**，帮你理解这个项目：它像一家「**天气预报橱窗 + 会员气象档案室**」——路人可以看各地天气；注册会员可以登记自己的传感器，把温湿度等数据存进数据库，并在网页上查看。

---

## 一、整体比喻：这家店在做什么？

| 现实比喻 | 在系统里对应什么 |
|----------|------------------|
| **店面橱窗**（谁都能看） | 首页的「区域天气预报」：不登录也能搜城市、看实况和 7 日预报；预报下方还有 **7 天气温折线图**（最高温/最低温两条线） |
| **会员区的「仪器示波器」** | 设备详情页：近几日**实测温度**曲线（可选 3/7/14 天），以及最近 120 条的温湿压综合图 |
| **会员卡 + 档案柜** | 用户注册/登录：密码存在数据库里，登录后浏览器里有一份「会话」，像临时胸牌 |
| **你名下的测量仪编号** | 每台设备有一个 `device_token`（长串密钥），硬件上报时「报这个号」就知道数据属于哪台设备 |
| **记账本** | MySQL 里的 `readings` 表：每次上报记一行时间 + 温度 + 湿度 + 气压 |
| **气象台合作方** | Open-Meteo 免费接口：程序替你去网上问「某地现在天气怎样」 |
| **可选的「对讲机频道」** | MQTT：设备可以不走网页接口，而是往消息中间件发一条 JSON，后台线程收到后同样写入数据库 |

**技术一句话**：后端用 **Python + Flask** 提供网页和 JSON 接口；用 **MySQL** 存用户、设备、读数；公开天气用 **HTTP 调 Open-Meteo**；可选 **MQTT** 订阅设备消息。

---

## 二、程序入口：启动时发生了什么？

文件主要是 `app.py`。当你运行 `python app.py` 时，大致顺序是：

1. **`init_db()`**  
   连接数据库，若没有表就创建（用户表、设备表、读数表），并做一些「老版本升级」用的补丁（例如给设备表加 `user_id`、给用户表加是否管理员）。

2. **`start_mqtt_if_enabled()`**  
   若环境变量里打开了 MQTT，就起一个**后台线程**，一直监听主题上的消息；没开就打印一句提示然后跳过。

3. **`app.run(...)`**  
   启动 **Web 服务器**，默认监听端口（可用环境变量 `PORT` 改），浏览器访问就能看到页面和接口。

**比喻**：先**装修并检查仓库（数据库）**，再决定要不要**开通对讲机专线（MQTT）**，最后**开门营业（Flask 监听请求）**。

---

## 三、数据库里三张「抽屉」各装什么？

| 表名 | 装什么 | 比喻 |
|------|--------|------|
| `users` | 用户名、密码哈希、注册时间、是否管理员 | 会员名册（密码不是明文，是「不可逆的暗号摘要」） |
| `devices` | 设备名、`token`、创建时间、属于哪个用户 | 你名下的仪器台账，每台有唯一钥匙 `token` |
| `readings` | 某设备在某时刻的温度/湿度/气压、来源(http/mqtt) | 仪器每次打卡的流水账 |

表之间用**外键**连着：删了设备，这台设备的历史读数也会跟着删（像撤掉仪器时，专用记录本一并归档清理）。

## 四、`app.py` 里的函数分工（按模块）

下面按**在文件里出现的逻辑顺序**说明。不要求你一次背下来，需要查功能时可以当「字典」翻。

### 4.1 天气代码与网络小工具

- **`_wmo_label_zh(code)`**  
  把天气 API 返回的数字代码（WMO）翻成中文，例如「晴朗」「小雨」。没有代码或非法就显示「—」。  
  **比喻**：把「天气编号」翻译成「人话」。

- **`_http_get_json(url, timeout)`**  
  用标准库发 GET 请求，把返回的 JSON 解析成 Python 对象；超时或解析失败会抛错。  
  **比喻**：派一个跑腿去指定网址取回一封信，并拆开读成你能用的数据。

- **`_utc_naive()`**  
  返回当前 **UTC** 时间，精确到秒，去掉微秒；写入数据库时用统一时区，避免各地电脑时钟混乱。  
  **比喻**：全店统一用「格林尼治钟表」记账时间。

---

### 4.2 配置与数据库连接

- **`_safe_db_name(name)`**  
  检查数据库名字是否只含安全字符（字母数字下划线），防止拼接 SQL 时被滥用。  
  **比喻**：确认仓库门牌号没有奇怪符号，避免被人钻空子。

- **`_mysql_params(with_db)`**  
  组装连接 MySQL 用的参数字典（主机、端口、用户、密码、字符集、游标类型等）；`with_db=False` 时不指定库名（用于「先建库」）。  
  **比喻**：准备一串开仓库要用的钥匙和地址信息。

- **`get_conn()`**  
  用上面的参数连上**已指定**的数据库 `MYSQL_DATABASE`，返回连接。  
  **比喻**：打开你日常用的那个档案库房间。

- **`maybe_create_database()`**  
  若环境变量 `MYSQL_AUTOCREATE_DB=1`，且库名安全，就执行 `CREATE DATABASE IF NOT EXISTS`。  
  **比喻**：老板授权「没有库房就自己盖一间」时才自动建库。

---

### 4.3 表结构检查与迁移（升级旧库）

- **`_column_exists(cur, table, column)`**  
  查 `information_schema`：某表是否已有某列。  
  **比喻**：翻设计图纸看抽屉里有没有「隔层」。

- **`_constraint_exists(cur, table, name)`**  
  查某约束（如外键）是否已存在。  
  **比喻**：检查「设备必须挂在某个用户名下」这条规矩是否已经写在制度里。

- **`migrate_users_admin_column()`**  
  若 `users` 没有 `is_admin` 列就加上；再根据环境变量 `ADMIN_USERNAMES` 把对应用户设为管理员。  
  **比喻**：给会员册加「是否店长」一栏，并按名单提拔店长。

- **`migrate_devices_user_id()`**  
  旧库若设备没有 `user_id`：加列、把无主设备归到一个占位账号、设非空、加外键。  
  **比喻**：以前仪器没写主人，现在统一贴上「历史遗留」或真实主人标签。

- **`init_db()`**  
  可能建库 → 建三张核心表 → 跑上面两个迁移。启动时整库的**总装配**。  
  **比喻**：开业前清点并改造货架，保证今天用的规则一致。

---

### 4.4 核心业务：读数与设备归属

- **`save_reading(device_id, temperature, humidity, pressure, source)`**  
  往 `readings` 插入一行，带时间戳。用锁 `_db_lock` 减少并发时的小概率冲突。  
  **比喻**：给某台仪器今天的读数在流水账上记一笔；`source` 标记是「柜台登记(http)」还是「对讲机(mqtt)」。

- **`token_for_device_id(token)`**  
  用 `token` 查 `devices` 表，返回设备主键 `id`；没有则 `None`。  
  **比喻**：凭仪器钥匙牌查「这是几号仪器」。

- **`_device_owned(device_id, user_id)`**  
  查该设备是否属于该用户（普通权限用）。  
  **比喻**：确认这本台账是不是你本人柜子里的。

- **`_device_for_actor(device_id, user_id, is_admin)`**  
  普通用户只能看自己的设备；管理员可查任意设备，并多带出 `owner_username`。  
  **比喻**：普通会员只能开自己的储物柜；店长可以打开任意柜，并看到「柜主名字」。

- **`_reading_stats(device_id)`**  
  对某设备所有读数做统计：条数、温度均值/最小/最大、湿度均值、气压均值。  
  **比喻**：把这台仪器整本流水账做个「摘要统计」。

---

### 4.5 登录态（Session）

- **`current_user_id()`**  
  从 Flask `session` 里取当前登录用户 id；未登录为 `None`。  
  **比喻**：看你胸牌上有没有会员号。

- **`current_is_admin()`**  
  当前用户是否管理员。  
  **比喻**：胸牌上是否写着「店长」。

- **`login_required_redirect()`**  
  未登录则重定向到登录页，并带上 `next` 方便登录后跳回原来想去的地址。  
  **比喻**：进内区前查胸牌，没有就去前台登记。

---

### 4.6 公开天气 API（Open-Meteo）

- **`api_weather_search()`**（路由：`GET /api/weather/search`）  
  参数 `q`：城市关键词。转发到 Open-Meteo 地理编码接口，整理成前端好用的列表（名称、省/州、国家、经纬度、展示用 `label`）。  
  **比喻**：帮顾客在地球仪上「搜地名」，返回一排候选点。

- **`api_weather_current()`**（路由：`GET /api/weather/current`）  
  参数 `lat`、`lon`：调预报接口，取当前实况 + 7 日每日最高温/最低温/天气代码，并把天气代码译成中文。  
  **比喻**：选定经纬度后，问气象台「现在怎样、未来一周概况」。

---

### 4.7 认证相关页面

- **`register()`**（`/register`）  
  GET 显示注册页；POST 校验用户名格式、密码长度与两次一致，密码哈希后插入 `users`，成功后写入 `session` 并跳转首页。  
  **比喻**：办会员卡，暗号（密码）只存「摘要」不存明文。

- **`login()`**（`/login`）  
  校验用户名密码，`check_password_hash` 比对；成功则 `session` 里记下 `user_id`、`username`、`is_admin`；`next` 防开放重定向（必须以 `/` 开头）。  
  **比喻**：前台核对身份后发胸牌。

- **`logout()`**（`/logout`）  
  清空 `session`。  
  **比喻**：退卡交回胸牌。

---

### 4.8 页面与设备 API

- **`index()`**（`/`）  
  首页模板：公开天气区始终可用。若已登录，则查该用户（或管理员查全站）的设备列表，并对每台设备查**最新一条**读数，传给模板渲染。  
  **比喻**：橱窗照常展示天气；会员进来再多摆一排「我的仪器近况卡片」。

- **`device_detail(device_id)`**（`/device/<id>`）  
  需登录；用 `_device_for_actor` 鉴权；拉 `_reading_stats`；模板展示单台设备详情与统计。  
  **比喻**：打开某一台仪器的专属详情页。

- **`api_register_device()`**（`POST /api/devices`）  
  登录用户新建设备，生成随机 `token`，写入 `devices` 并绑定 `user_id`。  
  **比喻**：给你办一台新仪器，发一把新钥匙 `token`，以后上报都认这把钥匙。

- **`api_patch_device(device_id)`**（`PATCH /api/devices/<id>`）  
  改设备名；管理员可改任意设备，普通用户只能改自己的。  
  **比喻**：给仪器改昵称。

- **`api_delete_device(device_id)`**（`DELETE /api/devices/<id>`）  
  删除设备（级联删读数，由数据库外键保证）。  
  **比喻**：注销这台仪器及其流水账。

- **`api_ingest()`**（`POST /api/ingest`）  
  **硬件/脚本常用**：JSON 里带 `device_token`（或 `token`），校验后把温度/湿度/气压写入 `readings`，`source=http`。  
  **比喻**：邮筒投信——不用登录网页，只要令牌对，就记一笔。

- **`api_readings(device_id)`**（`GET /api/devices/<id>/readings`）  
  登录且有权后，返回读数列表，时间转成 ISO 字符串。支持两种取数方式：  
  - **不传 `days`**（和以前一样）：按「条数」取最新记录，`ORDER BY id DESC`，默认 `limit=200`，最大 **500**。  
  - **传 `days=1～30`**：按「时间窗口」取从 **当前 UTC** 起往回 `days` 天内的数据；在窗口内先按时间倒序取最多 `limit` 条（默认 **2000**，最大 **3000**），再按时间**正序**返回，方便画时间轴。  
  **比喻**：不传 `days` 像「翻日记本最后几页」；传了 `days` 像「把最近一周里写的页挑出来按日期排好」。

- **`api_export_csv(device_id)`**（`GET /api/devices/<id>/export.csv`）  
  同上鉴权，导出 CSV 附件。  
  **比喻**：把流水账复印成 Excel 能打开的表格带走。

- **`api_health()`**（`GET /api/health`）  
  连一下数据库执行 `SELECT VERSION()`，用于运维探活。  
  **比喻**：拍一下仓库门铃，听有没有回声。

---

### 4.9 MQTT（可选）

- **`start_mqtt_if_enabled()`**  
  若 `MQTT_ENABLE=1` 且安装了 `paho-mqtt`：创建客户端，订阅 `weather/+/data`（`+` 是通配符，中间一段当作 **token**）。收到消息后解析 JSON，调用 `save_reading(..., source="mqtt")`。在单独**守护线程**里 `loop_forever`。  
  **比喻**：在店里装一部**总机**，所有仪器的对讲机按「天气/令牌/数据」频道喊一声，总机记下来。

**主题格式约定**：`weather/<device_token>/data`，负载为 JSON，可含 `temperature`、`humidity`、`pressure` 等字段。

## 五、前端模板大概在干什么？（不写代码细节）

- **`templates/base.html`**：全站公共外壳（导航、样式、脚本引用）；预留 **`{% block head_extra %}`**，子页面可插入额外脚本（如图表库）。像**统一的店招和菜单栏**。
- **`templates/index.html`**：首页——城市搜索、调用 `/api/weather/*` 拉实况与 **7 日 `daily` 数据**；在「未来 7 天」格子下方用 **Chart.js**（CDN）画 **「7 天气温曲线」**（红线最高温、蓝线最低温）；登录区显示设备卡片、添加设备等。像**橱窗 + 会员区**。切换城市时会销毁旧图表再画新的，避免叠图。
- **`templates/device.html`**：单设备详情；通过 **`{% block head_extra %}`** 引入 Chart.js。页面包含两块图：  
  1. **近几日温度变化**：请求 **`/api/devices/<id>/readings?days=3|7|14&limit=2000`**，主要画**温度**；点太多时前端会**抽样**显示，避免卡顿。  
  2. **最近读数（温 / 湿 / 压）**：请求 **`?limit=120`**（不传 `days`），三条线同图。  
  像**单台仪器的专页 + 示波器**。
- **`templates/login.html` / `register.html`**：表单提交到对应路由。像**登记台**。

前端通过浏览器 **fetch** 调后端 JSON 接口，页面不刷新也能更新部分内容（你打开开发者工具「网络」面板会看到请求）。

### 5.1 两种「温度曲线」别搞混（小白常问）

| 在哪里 | 数据从哪来 | 曲线含义 |
|--------|------------|----------|
| **首页**「7 天气温曲线」 | Open-Meteo 的 **预报** `daily`（`tmax` / `tmin`） | 未来约 7 天，**每天一个最高温、一个最低温**——是气象台模型算出来的，不是你的传感器 |
| **设备详情页**「近几日温度变化」 | 你自己设备写入 MySQL 的 **`readings`** | 过去若干天里**真实上报**的温度，可选 3/7/14 天窗口 |

**比喻**：首页图像是**电视台播的一周天气走势**；设备页图像是**你家温度计自己记的一周日记**。

## 六、依赖包各自像什么？

| 包 | 作用 | 比喻 |
|----|------|------|
| **Flask** | Web 框架：路由、模板、session、返回 JSON/文件 | 店面接待流程与收银系统 |
| **PyMySQL** | 连接 MySQL 执行 SQL | 与档案库的管道 |
| **werkzeug.security** | 密码哈希与校验 | 刻印章、验印章，不存明文密码 |
| **paho-mqtt**（可选） | MQTT 客户端 | 对讲机客户端 |
| **cryptography** | 常作为 PyMySQL 连接 SSL 等场景的依赖 | 运输途中的加固件（间接需要） |

首页与设备页的折线图使用 **Chart.js 4.x**，通过 **CDN**（`cdn.jsdelivr.net`）在模板里 `<script>` 引入，**不写在 `requirements.txt` 里**（那是给 Python 用的）。画图需要能访问该 CDN；若完全离线部署，需改为本地静态文件引用。

## 七、环境变量速查（和程序行为的关系）

| 变量 | 含义（简述） |
|------|----------------|
| `MYSQL_HOST` / `MYSQL_PORT` / `MYSQL_USER` / `MYSQL_PASSWORD` / `MYSQL_DATABASE` | 数据库连接信息 |
| `MYSQL_AUTOCREATE_DB=1` | 允许程序自动建库（库名须安全） |
| `SECRET_KEY` | Flask 会话加密用，生产环境务必设置长随机串 |
| `ADMIN_USERNAMES` | 逗号分隔用户名，启动时把匹配用户设为管理员 |
| `MQTT_ENABLE` / `MQTT_BROKER` / `MQTT_PORT` | 是否启用 MQTT 及 broker 地址 |
| `PORT` | Web 监听端口，默认 5000 |
| `FLASK_DEBUG_RELOAD` | 与是否开启自动重载有关（开了 MQTT 时常关重载避免双线程） |

## 八、把一条数据从传感器到网页画出来（串联比喻）

1. 你在网页登录，点「添加设备」→ 后端 **`api_register_device`** 生成 **`token`**，写入 **`devices`**。  
2. 树莓派/单片机定时向 **`POST /api/ingest`** 发 JSON（带 `device_token`），或往 MQTT 主题发消息。  
3. 后端 **`api_ingest`** 或 MQTT 回调里 **`token_for_device_id`** 找到设备 id，再 **`save_reading`** 插入 **`readings`**。  
4. 你刷新首页 **`index`**：后端查出设备列表和每台 **`readings` 最新一条**，模板画成卡片。  
5. 点进某台设备：**`device_detail`**；前端用 **`/api/devices/<id>/readings?days=…`** 画**近几日实测温度**曲线，用 **`?limit=120`** 画最近条数的温湿压图。

**一句话**：**令牌认设备 → 流水账记读数 → 网页按权限查库展示**；首页再多问一句 Open-Meteo，把**预报气温**画成 7 日折线。
